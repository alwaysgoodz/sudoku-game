### 2. 可能的问题和标准答案

#### 关于OOP概念

**Q: 请解释你的项目如何体现了封装？**
A: 
- 所有类的字段都使用private修饰符
- 通过public的getter/setter方法控制访问
- 例如SudokuPuzzle类中的grid字段是protected，只能被子类访问
- 数据库连接在DatabaseManager中被封装，外部只能通过公共方法访问

**Q: 继承在你的项目中是如何实现的？**
A:
- SudokuPuzzle继承自抽象类Puzzle
- Puzzle提供了通用的谜题功能，如getGrid()、setValue()等
- SudokuPuzzle实现了特定于数独的逻辑，如isValidMove()
- 这展示了"is-a"关系：SudokuPuzzle是一种Puzzle

**Q: 多态性如何在项目中体现？**
A:
- SudokuPuzzle实现了Solvable接口
- 可以通过Solvable接口引用SudokuPuzzle对象
- 运行时根据实际对象类型调用相应的方法
- 例如：Solvable puzzle = new SudokuPuzzle(); puzzle.solve();

**Q: 抽象是如何应用的？**
A:
- Puzzle抽象类定义了所有谜题的通用行为
- Solvable接口抽象了可解决谜题的行为
- 隐藏了复杂的实现细节，只暴露必要的接口

#### 关于设计模式

**Q: 你使用了哪些设计模式？**
A:
1. **单例模式**: DatabaseManager确保全局只有一个数据库连接实例
2. **MVC模式**: 分离了模型(SudokuPuzzle)、视图(SudokuGamePanel)和控制器(GameController)
3. **DAO模式**: GameDAO封装了所有数据库操作
4. **观察者模式**: GUI组件监听用户输入事件

**Q: 为什么选择单例模式管理数据库连接？**
A:
- 确保应用程序中只有一个数据库连接实例
- 避免资源浪费和连接冲突
- 提供全局访问点
- 便于管理数据库连接的生命周期

#### 关于数据库

**Q: 为什么选择Apache Derby？**
A:
- 项目要求使用Apache Derby
- 嵌入式数据库，无需额外安装和配置
- 纯Java实现，跨平台兼容
- 适合小型应用程序

**Q: 你的数据库设计思路是什么？**
A:
- 三个主要表对应不同的业务需求
- game_history记录完成的游戏
- player_stats聚合玩家统计信息
- saved_games保存游戏进度
- 使用外键关系确保数据一致性

**Q: 如何处理数据库异常？**
A:
- 使用try-catch块捕获SQLException
- 自定义SudokuException提供用户友好的错误信息
- 事务管理确保数据一致性
- 连接池管理避免资源泄露

#### 关于测试

**Q: 你的测试策略是什么？**
A:
- 使用JUnit 5进行单元测试
- 测试覆盖核心业务逻辑和数据库操作
- SudokuPuzzleTest测试游戏逻辑
- GameDAOTest测试数据持久化
- 每个测试方法都有明确的测试目标

**Q: 测试中遇到的主要挑战？**
A:
- 数据库测试需要清理测试数据
- 使用@BeforeEach和@AfterEach确保测试独立性
- 模拟复杂的游戏状态进行边界测试
